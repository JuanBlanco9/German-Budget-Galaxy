WHAT IT IS
----------
Interactive visualization of TOTAL PUBLIC SPENDING for 4 countries (Germany, USA, France, UK).
Not just central government budgets — includes social insurance, states/regions, municipalities.
Users can zoom into individual budget items, compare countries side-by-side, explore spending evolution.

Live: https://budgetgalaxy.com
Repo: https://github.com/JuanBlanco9/Budget-Galaxy
Server: 96.30.199.112 (Vultr Atlanta, Ubuntu 24.04)

TOTAL SPENDING COVERED (2024)

STACK
-----
- Frontend: Single HTML file (~470KB) at frontend/index.html
  - D3.js v7 (circle packing, semantic zoom)
  - Chart.js 4 (Budget Evolution line charts)
  - Vanilla JS, no framework
- Backend: FastAPI + Python at api/main.py (~200 lines)
  - Serves static files + JSON API
  - Runs on uvicorn port 8088, nginx reverse proxy + Let's Encrypt SSL
- Data: JSON tree files in data/ directory (no database)

TRANSLATION SYSTEM
------------------
- galaxyTranslations{} - Inline dict (~200 entries) for exact name matches
- translations_new_branches.json - 306 entries loaded via fetch, merged at startup
- galaxyWordMap{} - 565 word-level DE→EN translations for Titel-level names
- trG(name) - Main translation function: exact match → wordMap → cache
  - Sorts wordMap by length desc to prevent compound word breakage
  - Uses Unicode-aware boundaries (lookbehind with umlauts)
  - loadGalaxy() awaits translation fetch before rendering
- trFR(name) - French translation from translations_fr_en.json
- frTranslations{} - Loaded from JSON, 350+ entries

ENRICHMENT SYSTEM
-----------------
Two systems coexist:
1. ID-based (DE Bundeshaushalt only): enrichedNodes{} in HTML, keyed by EPL/Kapitel ID
   - Rich data: legal basis, evolution, beneficiaries, controversy, source
   - Covers depth 1-2 (~75 entries)

2. Name-based (all countries): PROGRAM_ENRICHMENT{} loaded from per-country JSONs
   - Format: {name: {y: year, d: 'description', b: 'beneficiaries'}}
   - DE loads 3 files: sozialversicherung + laender + program_enrichment
   - FR loads 2 files: program_enrichment + securite_sociale_enrichment
   - UK/US load 1 file each

INTEGRATION SCRIPT
------------------
scripts/integrate_all_branches.js merges new branches into existing trees.
  node scripts/integrate_all_branches.js [--dry-run] [de|uk|fr|all]

It:
1. Reads SV data (gkv, rente, ba, pflege, dguv) + Länder + Kommunen JSONs
2. Adds them as children of the root node in each bundeshaushalt_tree_YYYY.json
3. Same for UK (adds Local Authorities) and FR (adds Sécurité Sociale)
4. Updates root name ("Germany Public Spending 2024") and recalculates root value
5. Adds _disclaimer fields from DATA_LIMITATIONS.json
6. Backs up originals to data/backups/

  PHASE 2 — Council Spend > £500 supplier metadata (4 councils, METADATA only)

    KEY DESIGN DECISION: Spend > £500 transparency data is a SEPARATE accounting
    view from MHCLG net current expenditure. It includes capital expenditure,
    transfers, and payroll, but EXCLUDES sub-£500 payments. The two totals
    DO NOT RECONCILE and CANNOT be inserted as tree children that sum to the
    MHCLG service value. Instead, we attach `_top_suppliers` METADATA to each
    service node — the frontend renders it as an info panel below the tree
    hierarchy with a clear coverage_note explaining the discrepancy.

    Why this matters: a previous instinct would have been to inject suppliers
    as tree children, but that would have lied about the numbers. The metadata
    approach is more honest and the frontend panel is clearly labeled as
    "Top suppliers (transparency disclosure)" — separate visual treatment from

OSCAR 2023 NHS DATA QUALITY FIX (April 10, 2026) — FORENSIC INVESTIGATION
--------------------------------------------------------------------------
SYMPTOM
  After applying v13 NHS netting to both 2023 and 2024, the YoY comparison
  showed UK 2023→2024 dropping by £106B (£1,486B → £1,380B). UK total managed
  expenditure should be growing year over year, not shrinking. The drop was
  spurious and concentrated almost entirely in DHSC.

INVESTIGATION
  1. Compared DHSC composition 2023 vs 2024:
     - 2023: DHSC = £210.1B, NHS Trusts (residual) = £114.5B
     - 2024: DHSC =  £83.4B, NHS Trusts (residual) =   £4.7B
  2. The £4.7B residual in 2024 represents capital + non-EXP0390 items
     (TAC EXP0390 captures ~96% of DHSC NHS Trusts allocation). The £114.5B
     in 2023 was 24x larger than expected, suggesting an additional source
     of double-counting that v13 netting did not catch.
  3. Walked the pre-netting backup tree for 2023 DHSC > NHS Trusts:
       £238.69B  NHS Trusts (parent)
         £124.62B  7.A Medical services
           £123.04B  Other NHS Trust Spending
           £   6.62B  NHS North East and North Cumbria ICB
           ... (42 ICBs, sum = £115.65B)
         £114.07B  n/a > Non-patient-facing NHS expenditure
  4. Found TWO levels of structural problems in OSCAR 2023:
     (a) Internal bug in 7.A Medical services: parent value £124.62B but
         children sum to £238.69B. Caused by old enrich_uk_nhs.js injecting
         42 ICBs ALONGSIDE "Other NHS Trust Spending" instead of replacing it.
     (b) "Non-patient-facing NHS expenditure" line £114.07B inside NHS Trusts
         that does NOT exist in OSCAR 2024 schema.
  5. Compared OSCAR 2023 vs OSCAR 2024 DHSC > NHS Trusts structure side-by-side
     using both pre-netting backups:
       OSCAR 2023: NHS Trusts = £238.69B
         > 7.A Medical services £124.62B + Non-patient-facing £114.07B
       OSCAR 2024: NHS Trusts = £135.54B
         > 42 ICBs (clean) + Other NHS Trust Spending. NO Non-patient-facing line.
     This is evidence that HM Treasury cleaned up the OSCAR schema between
     the 2023 and 2024 release, eliminating the "Non-patient-facing" bucket.

CROSS-REFERENCED GROUND TRUTH (independent sources)
  - DHSC Annual Report 2022-23: total resource outturn £177.1B
    https://www.gov.uk/government/publications/dhsc-annual-report-and-accounts-2022-to-2023
  - HM Treasury PSA July 2023: DHSC total DEL £181.7B
    https://www.gov.uk/government/statistics/public-spending-statistics-release-july-2023/public-spending-statistics-july-2023
  - NHS England Annual Report 2022-23: total expenditure £155.1B
    https://www.gov.uk/government/publications/nhs-england-annual-report-and-accounts-2022-to-2023
  Our pre-fix OSCAR 2023 DHSC = £334.3B was nearly 2x the published figure.
  The £152.6B overshoot is approximately the size of NHS England commissioning
  (£153B), confirming cross-node double-counting was being captured at both
  the commissioner (NHS England) and provider (NHS Trusts) levels.

FIX APPLIED — scripts/fix_oscar_2023_nhs.js
  Reduced DHSC > NHS Trusts (residual after v13 netting) by exactly £114.07B,
  the documented value of the Non-patient-facing line. Sets _oscar_2023_fix=true
  on the node with full provenance metadata (_oscar_2023_fix_amount,
  _oscar_2023_fix_note explaining what was removed and why).

  Pre-fix:  DHSC £210.08B, NHS Trusts £114.47B, root £1,486.56B
  Post-fix: DHSC  £96.01B, NHS Trusts  £0.41B, root £1,372.49B

  Reconciliation:
    NHS Trusts residual £0.405B / TAC EXP0390 £124.22B = 0.3% overhead
    2024 reference:     £4.700B / TAC EXP0390 £130.84B = 3.6% overhead
  Both years have residual ratios consistent with capital + non-EXP0390 items,
  though 2023's 0.3% is suspiciously low (see KNOWN LIMITATION below).

YOY VALIDATION POST-FIX
  2022 → 2023: £1,257.6B → £1,372.5B (+£114.9B, post-COVID recovery)
  2023 → 2024: £1,372.5B → £1,380.5B (+£8.0B, steady growth)
  vs PRE-FIX:
  2023 → 2024: £1,486.6B → £1,380.5B (-£106.0B, spurious drop)

KNOWN LIMITATION
  The 0.3% (2023) vs 3.6% (2024) residual ratio suggests the 2023 residual
  should probably be £4-5B rather than £0.4B. We can only know the exact value
  by comparing against TAC 2022/23 capital expenditure data, which we don't
  currently have. The fix is therefore conservative: we removed exactly the
  documented £114.07B Non-patient-facing line, no more, no less. We did NOT
  invent a synthetic adjustment to match the 2024 ratio. This limitation is
  documented in the _oscar_2023_fix_note metadata on the node and in
  data/uk/intergovernmental_uk_2023.json oscar_2023_data_quality_fix section.

WHAT THIS TELLS US ABOUT HM TREASURY OSCAR
  OSCAR is not a single static schema. HM Treasury revises the structure
  between annual releases. OSCAR 2023 had a "Non-patient-facing NHS expenditure"
  bucket inside NHS Trusts that essentially captured cross-flows from NHS
  England commissioning, creating systematic double-counting. OSCAR 2024
  removed this bucket entirely. Future years should be checked for similar
  schema-level changes by comparing the same dept's structure across releases.

LESSONS LEARNED (April 7, 2026)
---------------------------------
1. NEVER modify tree JSON files on disk to restructure them. The Multiverse is extremely
   fragile to structural changes (negative values, zero-value parents, circular refs).
   Instead, restructure IN MEMORY in the frontend JavaScript using async functions like
   _loadUSCombinedTree() and _loadUKCombinedTree(). This keeps source files untouched
   and allows the catch() to prevent full Multiverse crashes.

2. OSCAR UK data contains negative accounting values (e.g. -£37B Export Credits).
   Clamping negatives to 0 creates zero-value parent nodes with children, which
   d3.pack() CANNOT handle (produces NaN radii → black screen). The correct fix is
   to either: (a) remove zero-value leaf nodes after clamping, OR (b) never modify
   the source files and handle in-memory only.

3. After ANY frontend change, always validate JS syntax before deploying:
   node -e "const h=require('fs').readFileSync('frontend/index.html','utf8');
   const s=h.indexOf('<script>');const e=h.lastIndexOf('</script>');
   try{new Function(h.slice(s+8,e));console.log('OK')}catch(e){console.log('ERR:'+e.message)}"

4. Always keep a backup of the last working frontend:
   data/backups/index.html.working.YYYY-MM-DD
   If something breaks, restore immediately: cp backup frontend/index.html && scp deploy

5. When re-applying changes after a revert, apply incrementally:
   a. Start from the last working version (git show HEAD:frontend/index.html)
   b. Apply HTML-only changes first (disclaimers, About section) — deploy and test
   c. Then apply JS changes one at a time — deploy and test each
   d. The Multiverse fetch block is the most dangerous — save it for last

6. const declarations at module scope CANNOT be duplicated (e.g. const REGION_COLORS
   existed in old code AND in new USA States code → SyntaxError killed entire page).
   Always grep for existing variable names before declaring new ones.

7. For subnational data integration pattern:
   a. Parse source data into standalone tree JSONs (data/{cc}/states/ or devolved/)
   b. Create _load{CC}CombinedTree() async function in frontend
   c. Fetch original tree + subnational tree in parallel
   d. Combine in memory: spread original children + add subnational sphere
   e. Use _net_value fields when deducting intergovernmental transfers
   f. Never wrap original departments in a container sphere — keep them sueltos

8. FORENSIC DEBUGGING PROCESS for source data quality issues (April 10, 2026):
   When YoY values move in unexpected directions, do NOT assume the source data is
   correct just because it's "official". Government accounting databases (OSCAR, PESA)
   change schema between annual releases. Process:
   a. Compute YoY deltas for the suspect metric across all available years
   b. Identify which year(s) break the expected trend
   c. Inspect the pre-modification backups to see the raw structure
   d. Compare structural differences node-by-node between the broken year and a
      known-good year (we did this for OSCAR 2023 vs 2024)
   e. Cross-reference the suspect total against INDEPENDENT published figures
      (annual reports, treasury statistics) — never trust a single source
   f. When you find a discrepancy, prefer SURGICAL deductions of documented
      line items over scaled adjustments. Document the exact value removed and
      the source it came from.
   g. Compute a sanity-check ratio (e.g., residual / TAC) and compare across years.
      If the ratios don't match, document the discrepancy honestly rather than
      adjusting the fix to make them match.
   h. Save full provenance metadata on the modified node (_fix_amount, _fix_note,
      _original_value) so future sessions can audit and undo if needed.
   See: scripts/fix_oscar_2023_nhs.js for a worked example.

9. NEVER invent numbers to make ratios match across years. If 2023 residual is 0.3%
   and 2024 is 3.6%, do NOT scale 2023 up to match. Document the discrepancy as a
   known limitation. The user explicitly values honest disclosure of imperfect fixes
   over polished but synthetic numbers. (Reinforces feedback_never_approximate_data.md.)

10. METADATA vs TREE CHILDREN — when integrating two datasets that don't reconcile.
    Some data sources count things differently and their totals don't match the parent
    node value. Examples:
      - Council Spend > £500 vs MHCLG net current expenditure (different basis)
      - DHSC L5 supplier list (£137.9B gross) vs DHSC node post-netting (£83.4B)
      - NHS Workforce composition (WTE) vs NHS staff costs (£) — same money, two lenses
    When this happens, do NOT inject as tree children that "should" sum to the parent —
    that would force you to either:
      (a) lie about the numbers (scale to fit), or
      (b) leave the tree integrity broken
    Instead, attach as `_top_suppliers` / `_workforce` / `_<custom>` METADATA on the
    node, with a `coverage_note` field that explicitly explains why the totals differ.
    The frontend renders metadata as a separate visual panel (not as nested tree
    children) so users immediately see it's a different accounting view.
    Examples in the codebase:
      - renderCouncilSuppliersPanel() — Council Spend > £500
      - renderNHSStaffPanel() — NHS Workforce + cost types
      - DHSC._top_suppliers — NHS L5 commissioning view
    The teal sidebar color and "(transparency disclosure)" labeling distinguishes
    metadata panels from real tree children visually.

11. HEADER AUTO-DETECT for inconsistent CSV files. Government CSVs often have
    inconsistent header positions: some files have a grand-total prefix row, some
    have publication metadata, some are clean. Hardcoding `headerRowIdx: N` breaks
    when even one file in the batch differs.
    Solution: pass a `headerHint` string and search for it in the first 8 lines of
    each file. Example: Manchester monthly CSVs have headers at row 2 in apr_2023
    and may_2023 (with grand total prefix), but row 0 in the other 10 months. The
    auto-detect found "Body Name" string and worked across all 12 files.
    This is now the default pattern for processMonthlyCouncil() in
    build_council_spend_lookup.js.

CRITICAL DATA RULE
-------------------
NEVER add data without a verified, downloaded, machine-readable official source.
Every number must be traceable to a specific dataset. Do NOT use approximate figures.
If no machine-readable source exists, say so and don't fabricate.
