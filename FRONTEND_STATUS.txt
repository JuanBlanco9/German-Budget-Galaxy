======================================================
OPINIÓN HONESTA — QUÉ FUNCIONA Y QUÉ NO
======================================================

LO QUE FUNCIONA BIEN:
  - El Galaxy view es espectacular. El circle packing con zoom semántico
    es intuitivo y visualmente impactante. Es el hook que engancha.
  - El Multiverse para comparar países es único — no conozco otra tool
    pública que haga esto con datos reales de 4 países.
  - La profundidad de datos UK es impresionante: 206 NHS trusts con
    cost breakdown, 401 councils con servicios, 19 councils con suppliers.
  - Los paneles de suppliers (renderCouncilSuppliersPanel) son claros
    y bien etiquetados con disclaimers honestos sobre reconciliación.

LO QUE NO FUNCIONA O PODRÍA MEJORAR:

  1. SINGLE FILE DE 500KB
     El index.html tiene 173 funciones, 4 sistemas de traducción,
     5 loaders de datos, y múltiples tab systems en un solo archivo.
     Es mantenible ahora pero frágil: un typo en cualquier función
     rompe todo el sitio. No es urgente pero eventualmente necesita
     modularización.

  2. DATOS INVISIBLES ($108 subcategorías + geo no se muestran)
     Invertimos horas clasificando subcategorías y enriqueciendo con
     Companies House. Los datos están en el JSON pero NADA en el
     frontend los muestra. Es la mejora de mayor impacto inmediato.

  3. MOBILE EXPERIENCE
     No está testeado en móviles reales. El Galaxy con D3 SVG
     probablemente funciona mal en pantallas < 768px. El sidebar
     de info compite con el canvas por espacio. El Multiverse es
     probablemente inutilizable en mobile.

  4. NO HAY SEARCH GLOBAL
     Un usuario que quiere encontrar "Kier Group" o "Serco" tiene
     que navegar manualmente hasta el council correcto y el servicio
     correcto. Con 3,600+ suppliers enriquecidos, una búsqueda
     cross-council sería tremendamente útil.

  5. BUDGET EXPLORER ES FUNCIONAL PERO ABURRIDO
     Es una lista jerárquica estándar. No tiene el wow-factor del
     Galaxy. Podría beneficiarse de treemaps, sankey diagrams, o
     al menos sparklines inline.

  6. EVOLUTION TAB TIENE GAPS VISUALES
     Los datos de evolución tienen discontinuidades (OSCAR I→II,
     COVID spike) que se ven como errores visuales. Necesita mejor
     anotación de eventos históricos en el gráfico.

  7. ABOUT PAGE ES TEXTO DENSO
     La sección de methodology es completa pero es un wall of text.
     Necesita accordion/collapse, maybe infográficos sobre el pipeline
     de datos.

WHAT I WOULD NOT DO
======================================================

- NO agregar React/Vue/Svelte. El vanilla JS funciona y agregar
  un framework a un archivo de 500KB es una receta para desastre.
  Si se modulariza, hacerlo con ES modules puros.

- NO agregar una base de datos. Los JSON trees son simples, rápidos,
  y el total es ~15MB. Una DB agrega complejidad sin beneficio real
  a esta escala.

- NO agregar autenticación. Es un proyecto de transparencia pública.
  Todo debe ser abierto y accesible sin login.

- NO obsesionarse con Birmingham FY 2023-24. Los datos fueron
  purgados y no hay forma de recuperarlos. Procesamos FY 2024-25
  pero el nodo no existe en el tree (MHCLG submission missing
  post-S114). Esto se resuelve solo cuando MHCLG publique datos
  actualizados.

- NO agregar más councils hasta que el frontend muestre lo que
  ya tenemos. Tenemos 3,738 suppliers enriquecidos, 108 subcategorías,
  957 local spend flags — todo invisible. Mostrar eso primero.
