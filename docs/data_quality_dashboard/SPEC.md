# Data Quality Dashboard — Design Spec

> **Status:** design. Not implemented. Target audience: next Claude Code
> session(s) + human team. Philosophy document that drives what the
> dashboard must do and why.

## Purpose

Most gov-data dashboards hide their limitations. Budget Galaxy's differentiator
is transparency — every claim has a source, every gap has a reason. The Data
Quality Dashboard makes that philosophy **visible as a feature** instead of
buried in methodology prose.

**Who it's for:**
- Journalists evaluating whether to cite the data
- Researchers deciding reproducibility of analyses
- Policy people assessing limitations before action
- Other data providers auditing our methodology

**What it's NOT:**
- A marketing page
- A "we're doing great!" scoreboard
- An apology for what's missing

It's a diagnostic instrument. If something is 60% covered, the page shows
60% and explains the other 40%.

## Design principles

1. **Every number is clickable** — drill into raw source, into the methodology that computed it, into the issue tracker of who reported the caveat
2. **Every gap has £ impact** — coverage 86.5% is nothing without knowing it's £20B missing from a £141.7B universe
3. **Every caveat has a blocker class** — "code-solvable" vs "policy-blocked" vs "FOI-pending" guides prioritization
4. **Every freshness badge has a timestamp** — "last updated 47 days ago" not "updated recently"
5. **No green-lighting** — absence of red flag does not mean "perfect". It means "no known issue". Document the difference.

## URL structure

```
/quality                              — overview (hero + summary cards)
/quality/coverage                     — per-dataset coverage breakdown
/quality/coverage/uk-mhclg           — specific dataset detail
/quality/gaps                         — known gaps + prioritization
/quality/gaps/{gap_id}                — single gap with full context
/quality/freshness                    — when was everything last updated
/quality/sources                      — source citations audit
/quality/reproducibility              — pipeline version + build hashes
/quality/suppliers                    — supplier enrichment quality breakdown
/quality/issues                       — known data quality caveats
/quality/changelog                    — data revisions history
```

## Data model — what JSONs feed this

Single source file: `data/_quality/dashboard.json`. Generated by a new script
`scripts/build_quality_dashboard.py` that consolidates everything below.

### Top-level schema

```json
{
  "generated_at": "2026-04-17T15:30:00Z",
  "generator_version": "1.0.0",
  "pipeline_versions": {
    "enrichment": "1.1.0",
    "procurement": "1.2.0-draft",
    "scrapers_uk_councils": "2.3.1"
  },
  "datasets": [ /* per-dataset coverage entries */ ],
  "gaps": [ /* prioritized list of known gaps */ ],
  "freshness": { /* per-source timestamps + age */ },

  "supplier_enrichment_quality": { /* for the supplier tab */ },
  "reproducibility": { /* build metadata */ },
  "source_audit": { /* citation counts */ }
}
```

### Gap entry schema

```json
{
  "id": "uk_other_authorities_cloudflare",
  "title": "UK Other Authorities: PCCs / Fire / Combined blocked by Cloudflare",
  "dataset_id": "uk_mhclg_2024",
  "scope_gbp": 12_000_000_000,
  "affected_units": 78,
  "blocker_class": "code_solvable",
  "blocker_details": "Cloudflare anti-bot on ~70 council websites blocks scraper",
  "mitigation": "Deploy scraper via UK-based VPS (~€4/mo) to bypass geo-filtering",
  "expected_pp_recovery": 7.5,
  "priority": "high",
  "status": "pending",
  "eta": null,
  "external_references": [
    { "type": "infra_ticket", "url": "#", "label": "VPS provisioning ticket" }
  ],
  "documented_at": "2026-04-15T00:00:00Z"
}
```

Blocker classes:
- `code_solvable` — unblocks with engineering work
- `policy_blocked` — the council chose not to publish or FOI-only
- `data_quality_upstream` — publisher's CSV has gaps/corruptions
- `legal_exemption` — security / defence redactions
- `waiting_publication` — next update cycle pending
- `requires_manual_curation` — needs human-in-the-loop

### Freshness entry schema

```json
{
  "source": "companies_house_api",
  "last_successful_call": "2026-04-17T10:30:00Z",
  "age_days": 0,
  "status": "fresh",
  "health": "green",
  "note": null
}
```

Statuses: `fresh` (<7d), `recent` (<30d), `stale` (<90d), `outdated` (>90d), `broken`.

### Known issue schema

Issues are quality caveats inside an otherwise-covered dataset — classification
warnings, pass-through distortions, categorization ambiguity. Not the same as
gaps.

```json
{
  "id": "hillingdon_ndr_passthrough",
  "dataset_id": "uk_mhclg_2024",
  "severity": "medium",
  "title": "Hillingdon £965M Non-Domestic Rates is a pass-through, not spend",
  "description": "Business rates collected on behalf of HMRC, not discretionary council spend. Classified correctly as an expenditure line but semantically misleading when comparing to other councils.",
  "impact_gbp": 965_000_000,
  "affected_entities": ["hillingdon_london_borough"],
  "first_documented": "2026-04-01T00:00:00Z",
  "mitigation_applied": "flagged in tree node with passthrough=true",
  "mitigation_visible_in_ui": true,
  "still_needs_attention": false
}
```

Severities: `low` (cosmetic), `medium` (affects interpretation), `high` (affects totals), `critical` (makes data wrong).

## UI layout — page by page

### `/quality` — Hero overview

```
┌──────────────────────────────────────────────────────────────┐
│ Budget Galaxy — Data Quality                                  │
│                                                               │
│  86.54%         400 profiles         303 with               │
│  of £141.7B     enriched (CH)        procurement              │
│  MHCLG          85% high-conf        history                  │
│                                                               │
│  Last rebuild: 6 hours ago · Deterministic build: ✓           │
└──────────────────────────────────────────────────────────────┘

[Coverage by Dataset]                   [Known Gaps]
(table, clickable rows)                  (list, sorted by £)

[Freshness Status]                      [Reproducibility]
(timeline last 30 days)                  (build metadata)

[Source Citations Audit]                [Changelog]
(45,213 total; by type)                  (last 10 updates)
```

Hero shows 3-4 top-level KPIs. Each clickable to drill. Secondary cards below.

### `/quality/coverage` — Dataset breakdown

Full table of every dataset we publish, with:
- Name, scope, coverage %, £ covered, £ missing
- Green/yellow/red health chip
- Last update timestamp
- Methodology link per row

Filterable by country / dataset type / status.

### `/quality/gaps` — Prioritized gap list

The most important diagnostic view. Sortable by:
- Scope_gbp (biggest first)
- Priority (high → low)
- Blocker class (code-solvable surface first)
- ETA (closest first)

Each row expandable to show:
- Why it's blocked
- Proposed mitigation
- Expected +pp recovery
- History of attempts

Filter chips across the top:
- [Code-solvable] [Policy-blocked] [Data quality] [Legal] [Waiting] [Manual]

### `/quality/gaps/{gap_id}` — Single gap detail

Full context page. Example: `/quality/gaps/uk_other_authorities_cloudflare`

```
UK Other Authorities: Cloudflare-blocked

SCOPE
  £12B across 78 authorities (PCCs / Fire / Combined)

BLOCKER
  [code-solvable badge]
  Cloudflare anti-bot on ~70 council websites blocks our scraper.
  Filtering is geo-based — UK-residential IPs pass, our cloud IP
  (Vultr Netherlands) fails.

MITIGATION PATH
  Deploy scraper on UK-based VPS. Estimated cost: €4/mo.
  Expected coverage gain: +7.5 percentage points.

AFFECTED UNITS
  [expandable list of 78 authorities with £]

HISTORY
  2026-03-15  Gap first identified
  2026-04-10  Confirmed Cloudflare via 403 response headers
  2026-04-17  Documented in coverage report
  2026-__-__  VPS provisioned  [pending]

WHY IT'S NOT FIXED YET
  Budget constraint. €4/mo needs approval + account setup.
```

### `/quality/freshness` — When was everything updated

Timeline view:
- Horizontal axis: last 30 days
- Each data source: a row with dots for update events
- Colors: green (fresh), yellow (getting stale), red (stale)

Plus a table with:
- Source name
- Last successful update
- Age
- Expected update cadence
- Next expected update
- Health status

### `/quality/suppliers` — Supplier enrichment quality

Composition:
1. **Match confidence distribution** (bar chart): high/medium/low/suspicious
2. **Suspicious matches table** with "review" action (links to review tool)
3. **UBO resolution distribution** (pie + explanation of each bucket)
4. **Procurement coverage** (303 with timelines, 97 without)
5. **Known false positives** (acknowledged wrong matches, kept for audit)

### `/quality/issues` — Known quality caveats

List of the 28+ documented issues (Hillingdon NDR, Other Services >30%, etc.)
with:
- Severity chip
- Impact £
- Description
- Mitigation applied (if any)
- Visible in UI? (y/n checkbox)

### `/quality/reproducibility` — The receipts

- Pipeline version number
- Commit SHA
- Build hash
- "Reproduce this exact output" command
- Dependencies frozen list
- Deterministic claim + evidence (= last 3 builds produced identical hashes)

### `/quality/changelog` — Dataset revisions log

Not a git log — a **human-readable log of data revisions** with reason:

```
2026-04-17  UK Other Authorities: 6 councils added (Dorset re-scraped after site fix)
            +£142M coverage, +0.08pp
2026-04-15  Hillingdon: NDR pass-through classification flag added
            No £ impact; improves interpretation
2026-04-10  Birmingham: FY 23/24 marked "S114 bankruptcy — unavailable"
            -£XXX M from coverage total, flagged in UI
```

## Implementation path

### New script: `scripts/build_quality_dashboard.py`

Reads from existing outputs across the repo:
- Tree integrity from `data/uk/integrity_report.json` (if exists)
- Coverage report from `UK_UNIVERSE_REPORT_2026-04-17.md` (parse)
- Supplier enrichment stats from `data/suppliers/_manifest.json`
- Match confidence from `data/recipients/uk/supplier_match_confidence.jsonl`
- UBO resolution from `data/suppliers/_index.json`
- Procurement coverage from `data/suppliers_v2/_procurement_coverage_report.json`
- Freshness from file mtimes of key sources
- Pipeline versions from script headers

Writes:
- `data/_quality/dashboard.json` (master)
- `data/_quality/dashboard.html` (static renderer, simple)
- `data/_quality/gaps.json` (denormalized for gap detail pages)
- `data/_quality/freshness.json` (for freshness page)

### Known issues registry

`data/_quality/known_issues.json` — **hand-maintained**. Each entry = one
documented caveat. Build script reads it; humans curate it.

```json
[
  {
    "id": "hillingdon_ndr_passthrough",
    "dataset_id": "uk_mhclg_2024",
    ...
  }
]
```

This becomes the source of truth for the "Issues" page. Adding a new issue is
a PR with one JSON entry. Reviewable.

### Gaps registry

`data/_quality/gaps.json` — same pattern. Hand-curated with structured fields.
Update when:
- A new gap is identified
- An existing gap is fixed (mark as `status: resolved`)
- Priority changes

### Integration with frontend

New page template. Minimum viable:
- Server-side rendered HTML (no SPA needed) reading `dashboard.json`
- Sidebar with sub-pages
- Tailwind for styling (matches existing Budget Galaxy)

Advanced version (later):
- Client-side interactivity for filters/sort
- Live "last updated" clock
- Subscribe-to-updates webhook

### Deploy cadence

Dashboard regenerates:
- Automatically after every pipeline rebuild (nightly)
- On-demand via `python scripts/build_quality_dashboard.py`

Changelog entry auto-generated when dataset coverage changes by >0.1pp.

## Why this design matters

### For periodismo / academia

Before running a story using our data, a journalist can:
1. Check the gap list for their angle (defence? councils? specific year?)
2. Read the methodology notes for the specific number they'll cite
3. See when the data was last refreshed
4. Get the reproducibility hash to cite

That moves Budget Galaxy from "tool I used" to "tool I can defend in editorial review."

### For trust over time

Data projects die when gaps go undocumented and users discover them later.
They live when gaps are public from day one and fixes are tracked.

Showing "86.54% covered" alongside "£19B gap, prioritized triage" is
**stronger** than claiming 95% with asterisks. Smart users see through the
asterisks.

### For team maintenance

When a council changes its CSV format and a scraper breaks, the dashboard
surfaces it as `status: broken, last_successful: 12 days ago` immediately.
It's a monitoring layer disguised as a transparency feature.

This is the "monitoring + self-documentation" we were missing in the
maintenance concern that came up in the portfolio review.

## Anti-patterns to avoid

- **"We're 95% complete" without context** — meaningless without total size + what's in the 5%
- **"Last updated: recent"** — lazy. Show the actual timestamp.
- **Hiding issues under "methodology" PDFs** — must be searchable, filterable, linked to the data row
- **Promises without ETAs** — "will improve" is vapor. "Blocked on €4/mo VPS" is honest.
- **Mixing gaps and issues** — gaps = data missing, issues = data present but caveated. Different mental models.

## Scope for MVP (what ships first)

Minimum viable dashboard:

1. `/quality` overview page reading a dashboard.json (1 page, static HTML)
2. `data/_quality/dashboard.json` auto-generated from existing files
3. `data/_quality/known_issues.json` + `gaps.json` manually curated with
   initial content from the Apr 17 coverage report + this supplier enrichment work
4. Build script + gitignore'd dashboard.json (regenerated on each pipeline run)

~2 sessions of work. Everything else on this spec can follow iteratively.

## What this unlocks

- "Ask Claude" chat grounded in quality-aware context — the LLM can correctly
  say "we don't cover that council" instead of hallucinating
- Procurement / research customers can evaluate before subscribing
- Internal team has a single source of truth for "what's working / what isn't"
- Every research paper citing Budget Galaxy can reference a reproducibility hash
- Potential integration with tools like Project Shield for automated fresh-data
  verification

## Open questions for team alignment

1. Should gap / issue registries be public (visible) or only semi-public (link from methodology)? Lean public — maximum transparency.
2. Changelog granularity — per-pipeline-run or only when coverage delta >0.1pp? Lean latter to avoid noise.
3. Freshness thresholds — fresh=<7d, recent=<30d? Or stricter?
4. Should suspicious-match-review link out to a separate tool, or inline here?
5. Does "changelog" go under quality or under its own top-level nav?

**Summary**: the dashboard transforms the 86.54%/400-profiles/13.46%-gap reality
into a **product feature that compounds trust over time**. It's the layer
that makes Budget Galaxy's open-data philosophy legible to the users who decide
whether to cite it. Costs ~2 sessions to ship MVP; pays back every time someone
asks "can I trust this number?"
